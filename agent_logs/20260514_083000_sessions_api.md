# Session API Restructure — Implementation Log

**Date:** 2026-05-14

## Summary

The frontend (Phase 2, React + Vite) uses a session-based REST API that the backend did not implement. The backend had a single `POST /plan` endpoint that both created and streamed the agent run in one long-lived request. The frontend expected session creation and SSE streaming as separate requests, plus list/get/interrupt/respond endpoints.

Root cause: the Phase 1 plan was written before the frontend was inspected. The "Future Work" note in the plan deferred full session API to Phase 4; in practice the frontend already required it from day one.

## Previous State

| Route | Description |
|-------|-------------|
| `POST /plan` | Create + stream agent SSE in one request |
| `POST /plan/{id}/respond` | Inject answer `{answer}` |

SSE events: raw prefixed strings (`"status: ..."`, `"thinking: ..."`, `"tool_call: ..."`, etc.)

## Changes Made

### New package
- `internal/event/event.go` — typed SSE event structs (`Token`, `Tool`, `Question`, `Done`, `Error`) + `Encode()` helper

### Modified: `internal/server/sessions.go`
Added to `session` struct: `goal`, `status`, `outCh chan string` (buffer 512), `respChan`, `cancel context.CancelFunc`, `createdAt`. Added `sessionView` for JSON serialisation. New methods: `register`, `get`, `view`, `list`, `setStatus`.

### Modified: `internal/server/handlers.go`
Replaced `handlePlan` / `handleRespond` with 6 handlers:
- `handleCreateSession` — starts agent goroutine, returns `{id}` as JSON immediately
- `handleStreamSession` — reads from `session.outCh`, streams SSE
- `handleListSessions` / `handleGetSession` — REST reads
- `handleRespondToQuestion` — accepts `{question_id, answer}`; question_id is accepted but ignored (single pending question invariant)
- `handleInterruptSession` — calls `session.cancel()`

### Modified: `internal/server/server.go`
Routes updated to match frontend contract.

### Modified: `internal/agent/agent.go`
All `out <-` writes changed from raw strings to `event.Encode(event.Token{...})`. Terminal error/done events removed from agent — the goroutine wrapper in `handleCreateSession` owns those.

### Modified: `internal/tools/registry.go`
Tool status events changed from `"tool_call/done/error: ..."` strings to `event.Encode(event.Tool{...})`.

### Modified: `internal/tools/ask_user.go`
Question event changed from `"question: ..."` to `event.Encode(event.Question{id, text})`. Question ID generated with `crypto/rand` (8-byte hex). Frontend echoes the ID in respond requests; backend ignores it (only one question pending at a time).

### Modified: `internal/tools/finalize.go`
Raw itinerary string changed to `event.Token`.

## Final Route Table

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/sessions` | Create session, start agent, return `{id}` |
| `GET` | `/sessions` | List all in-memory sessions |
| `GET` | `/sessions/{id}` | Get single session |
| `GET` | `/sessions/{id}/stream` | SSE event stream |
| `POST` | `/sessions/{id}/respond` | Answer pending `ask_user` question |
| `POST` | `/sessions/{id}/interrupt` | Cancel running agent |
| `GET` | `/health` | Health check |

## SSE Event Format

All events are JSON objects on a `data:` line followed by a blank line (`\n\n`).

| Event | Shape |
|-------|-------|
| Text chunk | `{"type":"token","content":"..."}` |
| Tool status | `{"type":"tool","name":"...","status":"running"\|"done"\|"error"}` |
| Agent question | `{"type":"question","id":"...","text":"..."}` |
| Session done | `{"type":"done","session_id":"..."}` |
| Agent error | `{"type":"error","message":"..."}` |

## Session Lifecycle

```
POST /sessions      → session created (status: running), goroutine started
GET  /sessions/:id/stream → reads from outCh until closed
agent finishes      → status set to done/error, done event emitted, outCh closed
GET  /sessions/:id  → returns final status
```

Sessions are in-memory only (Phase 1). They persist for the server lifetime and are not cleaned up. Phase 2 will move them to Postgres.

## Suggestions for Follow-Up (do not implement)
