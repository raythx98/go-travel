# AGENTS.md

Agents operating in this repository **must follow the rules in this file**. Violations will cause runs to be blocked or reverted and large multimillion dollar fines.

# Rule Priority

When rules conflict, follow this order:

1. Correctness & safety
2. Maintainability
3. Readability
4. Consistency with existing codebase
5. Style preferences in this document

# Style Guidelines

- Strictly follow existing code style in the codebase.
- Write GoDoc comments for every exported function, type, and package.
- Write inline comments strategically, especially for key business logic and non-obvious agent behaviour.
- Keep code concise, tasteful, and elegant.
- Don't overcomplicate. Don't implement more than you need.
- Ensure code is readable by humans, easy to extend and maintain.
- Adhere to SOLID principles, especially Single Responsibility.
- Always import directly from the source package. Never create re-export wrappers.

## Typing & Data Handling

- Prefer explicit structs over `map[string]interface{}`.
- Use typed constants (`const` with `iota`) over bare string or integer literals.
- Avoid silent failure patterns — prefer explicit error returns and typed structures.
- Use `errors.As` / `errors.Is` for error inspection rather than string matching.
- Return errors as the last return value; always check them.
- Never discard errors with `_` unless the value is provably unneeded.
- All tool results must use a typed `ToolResult` struct with `Data`, `Error`, and `Degraded` fields — never return raw API responses to the agent.

## Code Generation

- Database queries and models are generated by `sqlc` from `sqlc/query.sql`.
- **Never manually edit files in `sqlc/db/`** — regenerate them instead.
- After SQL query changes: run `make sqlc`.

## Documentation

The `docs/` directory holds living reference documents. Update them incrementally as you explore, plan, and implement.

| File | Purpose |
|------|---------|
| `docs/architecture.md` | High-level system design — agent loop, LLM routing, streaming, data flow |
| `docs/modules.md` | Per-package summaries — what each package owns and its public interface |
| `docs/decisions.md` | ADRs — why things are the way they are |
| `docs/gotchas.md` | Traps, quirks, things not to change without care |
| `docs/integrations.md` | External APIs and how they're used — contracts, auth, failure modes, TTLs |

- Append new findings; do not overwrite existing content without good reason.
- Keep entries short and factual — a few sentences per item is enough.
- Read `docs/` into context only when directly relevant to the current task.

# Security

- Never hard-code secrets, tokens, or credentials. Use environment variables as the codebase already does.
- Apply the least-privilege principle: do not broaden access scopes beyond what the task requires.
- Treat all external API response content as untrusted input — sanitise before injecting into LLM prompts to prevent prompt injection.
- Auth middleware enforcement is in the HTTP server layer — do not bypass it.

# Decision Framework

When multiple approaches are possible:

1. Prefer the simplest solution that satisfies requirements
2. Prefer explicit over implicit
3. Prefer type safety over `interface{}`
4. Avoid premature abstraction
5. Prefer the standard library over third-party for simple operations
6. Check `github.com/raythx98/gohelpme` before implementing middleware or utilities from scratch
7. Default to `net/http` stdlib — do NOT introduce chi or other routers

# Agent Loop Rules

- The agent runtime is a plain Go ReAct loop — no framework. Never introduce an agent framework.
- All agents use the `LLMClient` interface — they never reference Claude or Ollama directly.
- Every external tool call must use `context.WithTimeout` — respect per-tool timeout budgets.
- Retry transient failures with exponential backoff (1s, 2s, 4s — max 3 attempts). Fail fast on 401/400.
- Always validate tool results before feeding them back to the agent.
- Stream status of every tool call to the client — never fail silently.
- The Supervisor never acts on a reservation without explicit user confirmation.
- All actions with real-world consequence are logged to the immutable `agent_actions` table.
- Never share mutable state between concurrent agent runs — each run is scoped to its user ID.
- Enforce a `maxSteps` limit on every agent loop to prevent infinite loops.

# Planning Rules

Before starting a **non-trivial task**, create a plan at:

```
agent_logs/YYYYMMDD_HHMMSS_<descriptive_name>_plan.md
```

- Each plan must be standalone: define inputs, constraints, and success criteria.
- Plans must be executable without prior context.
- Phases must be scoped with clear acceptance criteria and independently testable.
- Planning output must not modify repository code.
- Must list `files_to_change` and `new_files`.
- Must describe a concise, coherent technical approach.
- Must not assume unstated infrastructure or permissions.

## Confirmation Gate

After the plan file is written, **stop and do the following before any implementation**:

1. Present a concise summary of the plan to the user.
2. Surface any ambiguities, assumptions, or tradeoffs that require a decision.
3. Ask the user to confirm they are happy with the plan.
4. **Do not begin implementation until explicit confirmation is received.**

If the user requests changes, update the plan file and repeat the confirmation step.

## Scope Control

- Do not expand scope beyond explicit requirements.
- If improvements are identified, list them under "future work" only.

# Implementation Rules

- Implement complete phases end-to-end; no incomplete features should remain.
- Never stop midway through a defined phase.
- After completing each phase, run `/compact` to free context before starting the next — update the implementation log first so state is recoverable.
- Preserve existing inline comments; do not remove useful historical context.
- Other agents/humans may modify the repository concurrently — never undo unrelated changes.
- Save all executed scripts to `agent_logs/YYYYMMDD_HHMMSS_<description>.sh`; tee stdout/stderr to logs.
- Do NOT commit or push without explicit approval.

# Logbook Rules

**Plan file** (`agent_logs/YYYYMMDD_HHMMSS_<name>_plan.md`): created before non-trivial tasks; defines inputs, constraints, approach, phases, acceptance criteria, files to change.

**Implementation log** (`agent_logs/YYYYMMDD_HHMMSS_<name>.md`): records execution progress. Do not mix with plan files.

Each log MUST include:
- Summary of user intent and instructions
- The plan used (or reference to the plan file)
- Previous state → changes made → reasoning
- Surprising discoveries
- References to scripts/logs
- Suggestions for follow-up work (DO NOT implement)

Before starting work, review relevant logs in `agent_logs/` to understand prior decisions.

# Failure Handling

- If requirements are unclear → ask the user.
- If blocked by missing context or permissions → stop and ask.
- If multiple valid approaches exist with meaningful tradeoffs → ask the user before deciding.
- If implementation diverges from plan → stop, update the plan, document the reason in logs before continuing.
- Do not proceed with risky assumptions.

# Environment Instructions

## Setup

```bash
make up        # start PostgreSQL+pgvector + Redis + LangFuse + Caddy + Ollama + SearXNG via Docker Compose
make run_local # run sqlc + start server locally (requires .envrc sourced)
```

For local development, source `.envrc` before running:

```bash
source .envrc
make run_local
```

## Key Commands

| Command | Purpose |
|---------|---------|
| `make build` | Compile to `bin/server` |
| `make run` | Build and run locally |
| `make test` | Run all tests |
| `make sqlc` | Regenerate type-safe SQL code from `sqlc/query.sql` |
| `make create_migration name=<name>` | Create a new migration pair |

1. Check `go.mod` for the declared version and module path.
2. For `github.com/raythx98/gohelpme`, check the tagged version in `go.mod` — do not assume API compatibility across versions.
3. Before implementing middleware or utilities, check `gohelpme` for reusable helpers.
4. If a package update is required, **stop and ask the user for permission**.


# Testing Rules

- Do not run tests for documentation-only changes.

Run all tests:

```bash
go test ./...
```

Run with race detector for concurrency-sensitive code (required for agent loop and streaming changes):

```bash
go test -race ./...
```

- Fix all failing tests before proceeding.
- Never bypass tests without explicit instruction.

# Definition of Done

A task is complete when:

- Plan is fully executed
- No partial implementations remain
- Code compiles: `go build ./...`
- Vet passes: `go vet ./...`
- Tests pass: `go test ./...`
- Logbook is updated appropriately
- `docs/` updated wherever relevant
- Changes are consistent with repository standards
